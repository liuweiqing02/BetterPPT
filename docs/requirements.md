# BetterPPT 需求文档（PRD）V1.1

## 1. 项目背景

在“文档转 PPT”场景中，常见问题包括：

- 内容表达生硬，难以形成汇报叙事。
- 图表与关键信息在转换过程中丢失。
- 视觉风格与企业模板不一致，需大量人工返工。

BetterPPT 目标是构建一个参考驱动的智能 Agent 系统，实现从 PDF 文档到 PPT 的自动化生成与风格迁移。

## 2. 项目目标

- 用户必须上传 `PDF 文档` 与 `参考 PPT`，并选择输出长度挡位（精炼/适中/详细）。
- 系统自动决定最终页数、章节结构与页面内容分配。
- 对长文档支持可选提示词，结合 RAG 进行内容重点控制。
- 输出风格与参考 PPT 高相似，内容来源于上传文档。
- 最终生成可编辑的 `.pptx` 文件，支持在线预览与下载。

## 3. 术语定义

- 文档：用户上传的源 PDF 文件。
- 参考 PPT：用户上传的风格模板 PPTX 文件。
- 长度挡位：输出密度配置，包含精炼、适中、详细三档；后端枚举值分别为 `concise|balanced|detailed`。
- 编辑式生成：将“生成 PPT”转化为“对模板 PPT 的批量增删改”。
- 结构化页面：封面、目录、章节页、结论页、对比页等具有固定布局功能的页面类型。

## 4. 用户角色与使用流程

### 4.1 用户角色

- 业务汇报人员
- 咨询/研究人员
- 产品与运营人员

### 4.2 主流程

1. 上传 PDF 与参考 PPT（必填）。
2. 选择长度挡位（必选：精炼/适中/详细）。
3. 可选输入提示词（长文档场景下指定重点）。
4. 提交任务并查看处理进度。
5. 预览生成结果并下载 PPT。

## 5. 功能需求

### 5.1 任务创建与校验

- 前端必须校验两类文件是否齐全，不齐全不可提交。
- 文件格式限制：
  - 文档：PDF
  - 模板：PPT/PPTX（输出统一为 PPTX）
- 可配置文件大小上限、页数上限、并发提交限制。
- 后端二次校验文件角色：
  - `source_file_id` 必须为 `pdf_source`
  - `reference_file_id` 必须为 `ppt_reference`

### 5.2 文档理解与内容规划

- 抽取文档结构（标题层级、段落、图表、关键术语）。
- 基于长度挡位自动规划页数与内容密度。
- 长文档支持 RAG：
  - 根据用户提示词做检索增强。
  - 提升指定主题内容在输出中的权重。

### 5.3 长度挡位规则（MVP 可验收口径）

以 20-60 页输入 PDF 为基准，目标页数范围如下：

- `concise`（精炼）：8-12 页，强调结论与行动项。
- `balanced`（适中）：13-20 页，信息与可读性平衡。
- `detailed`（详细）：21-30 页，覆盖更多细节与论据。

跨挡位约束：

- 同一输入文档下，`page_count(concise) < page_count(balanced) < page_count(detailed)` 必须成立。
- 文本密度（内容页平均正文字符数）应随挡位递增。
- 关键章节覆盖率（问题/分析/结论）三档均不低于 90%。

### 5.4 提示词与 RAG 行为约定

- `user_prompt` 为空，`rag_enabled=false`：默认摘要与规划流程。
- `user_prompt` 非空，`rag_enabled=false`：提示词仅作为规划偏好，不触发检索增强。
- `user_prompt` 非空，`rag_enabled=true`：提示词参与检索增强并提升主题权重。
- `user_prompt` 为空，`rag_enabled=true`：系统自动从文档关键词构造检索查询。

### 5.5 多模态模板分析

- 使用 ViT 提取参考 PPT 各页 embedding。
- 采用层次聚类得到模板页面类型集合。
- 结合统一 LLM API（默认 `gpt-4.1-mini`）抽取每类页面的结构化信息：
  - 页面功能（封面、目录、章节、对比、总结等）
  - 布局 schema（文本区、图片区、图表区、占比关系）
- 将结构 schema 作为后续生成约束，提高视觉迁移保真度。

### 5.6 编辑式生成引擎

- 以参考 PPT 为基础执行批量增删改：
  - 替换文本内容
  - 插入/替换图片
  - 调整图文布局
  - 新增或删除页面
- 自我修正机制：
  - 检测溢出、遮挡、空白异常、错位等问题
  - 自动重排并进行二次修正

### 5.7 任务管理与结果交付

- 提供任务状态：`created`、`validating`、`queued`、`running`、`succeeded`、`failed`、`canceled`。
- 失败支持错误原因查看与重试。
- 成功任务支持在线预览与文件下载。

## 6. 非功能需求

### 6.1 性能与容量（MVP）

- 普通任务端到端耗时目标：P50 <= 5 分钟，P95 <= 10 分钟。
- 系统可用性目标：月度 >= 99.5%。
- 任务成功率目标（非用户取消）：>= 95%。
- 单租户默认并发提交阈值：3（可配置）。

### 6.2 稳定性

- 任务失败可恢复，支持幂等重试。
- 后端关键流程具备超时与降级策略。
- Worker 支持断点续跑（按步骤跳过已完成节点）。

### 6.3 安全与治理

- 文件隔离存储，按任务与用户进行访问控制。
- 上传文件执行白名单与恶意文件扫描（扩展名 + MIME + 扫描结果）。
- 下载链接使用短时效签名。
- 默认留存策略（可配置）：
  - 源文件：7 天
  - 结果文件：30 天
  - 任务审计日志：180 天
- 支持用户发起删除与到期自动清理。

### 6.4 可观测性

- 记录任务链路日志与关键耗时指标。
- 提供错误分类统计，支持后续模型与策略优化。
- 记录任务事件流用于前端进度展示与问题回溯。

## 7. 技术方案约束

### 7.1 技术栈

- 主要语言：Python
- 架构：前后端分离
- 数据库：MySQL + Redis
- 远程仓库：<https://github.com/liuweiqing02/BetterPPT.git>

### 7.2 运行与开发要求

- 本地运行必须使用虚拟环境（venv/conda）。
- 提供 `.env.example`，最少包含：
  - MySQL 连接配置
  - Redis 连接配置
  - 模型服务配置（必须包含以下变量）：
    - `LLM_API_BASE`（默认：`https://api.openai.com/v1`）
    - `LLM_API_KEY`
    - `LLM_MODEL`（默认：`gpt-4.1-mini`）
  - 对象存储配置（如有）

### 7.3 Agent Worker 技术约束

- 所有核心 Agent 必须由 LLM 驱动，并通过统一 API 客户端调用（OpenAI 兼容协议）。
- 本地模型仅用于辅助能力，不替代核心推理链路：
  - PPT 视觉 embedding（如 ViT）
  - OCR/版面检测（如需要）
  - 轻量规则分类器
- Worker 内部步骤最少包含以下 Agent，并要求结构化输入输出：
  - `DocumentParseAgent`：输入 PDF 解析结果，输出结构化章节与证据片段
  - `RagRetrieveAgent`：输入检索 query，输出命中片段与引用
  - `OutlinePlanAgent`：输入文档结构与挡位，输出页级大纲
  - `LayoutMapAgent`：输入模板 schema 与页级大纲，输出页面映射计划
  - `SlideGenerateAgent`：输入页面映射计划，输出可执行编辑指令
  - `SelfCorrectAgent`：输入生成结果与质检信号，输出修正指令
- 所有 Agent 输出必须可追踪（入库 `task_steps.output_json` 与 `task_events`），用于回放和审计。

## 8. 系统架构（逻辑）

- 前端：上传、参数选择、进度与结果展示。
- 后端 API：任务受理、鉴权、状态查询、结果预览地址下发。
- Agent 编排层：
  - 文档解析 Agent
  - 检索增强 Agent（RAG）
  - 大纲规划 Agent
  - 版式映射 Agent
  - 质量修正 Agent
- 存储层：
  - MySQL：用户、任务、文件元数据、模板分析、步骤与事件日志
  - Redis：任务队列、缓存、分布式锁、进度状态

## 9. 数据模型（与 TDD 对齐）

### 9.1 MySQL 核心表

- `users`：用户信息
- `files`：上传与产出文件元数据
- `tasks`：任务主表（状态、挡位、耗时、错误码）
- `task_steps`：任务步骤执行记录
- `task_events`：任务事件流
- `template_profiles`：模板聚类分析结果
- `template_page_schemas`：模板页面结构化 schema

### 9.2 Redis 使用

- `stream:tasks:pending`：任务队列（Stream + Consumer Group）
- `lock:task:{task_no}`：任务级分布式锁
- `task:progress:{task_no}`：任务状态短期缓存
- `task:events:{task_no}`：最近 N 条事件缓存
- `rate_limit:create_task:{user_id}`：限流计数

## 10. 验收标准（MVP）

- 必填项校验正确：未上传 PDF 或参考 PPT 不可提交。
- 三档长度挡位生效：同文档下页数单调递增，且落入对应挡位范围。
- 长文档提示词可影响内容取舍；当 `rag_enabled=true` 时可观察到检索命中差异。
- 生成 PPT 在视觉风格上接近参考 PPT（人工抽检通过率 >= 85%）。
- 任务状态可追踪，失败可重试，成功可预览可下载。
- 核心接口具备契约测试，状态机流转具备单元/集成测试。

## 11. 版本规划

- M1：基础上传、任务处理、PPT 导出与下载。
- M2：模板聚类与结构化 schema 抽取，补齐在线预览。
- M3：RAG 提示词增强、自我修正能力提升。
- M4：质量评估体系与企业模板库扩展。

## 12. 风险与应对

- 风险：复杂版式映射失败导致页面错位。  
  应对：引入布局校验与回退模板策略。
- 风险：长文档信息过载导致输出冗长。  
  应对：分层摘要与长度挡位约束联合控制。
- 风险：图表解析能力不足导致信息丢失。  
  应对：图表区域识别与兜底截图策略。
- 风险：模型调用不稳定导致任务波动。  
  应对：关键步骤超时熔断、重试与降级模型。
